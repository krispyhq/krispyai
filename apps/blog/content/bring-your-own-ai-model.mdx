Most chat platforms pick the AI for you. You get whatever model the vendor bundled, at whatever margin they added, and the line item shows up in a subscription bill somewhere between "seats" and "AI add-on." You're not choosing the model — you're choosing the product tier.

Krispy is different because it's open-source. The AI layer is a config line. Which means the question is actually yours to answer: Workers AI, OpenAI, or Anthropic — and when does each one make sense?

## The short version

If you're already [deploying Krispy on Cloudflare](/self-host-live-chat-cloudflare), start with Workers AI. It's the default, it costs nothing, and it requires no extra API key. If you hit a real capability gap — tone, context length, domain-specific reasoning — swap to OpenAI or Anthropic in five minutes. You're not locked into anything.

Here's the full picture.

## Cloudflare Workers AI: the zero-friction starting point

Workers AI runs inside the same Cloudflare runtime as the rest of Krispy. No external API call, no separate bill, no key to rotate. The `AI` binding is declared in `wrangler.toml` and that's it.

```toml
# wrangler.toml
[ai]
binding = "AI"
```

```env
AI_PROVIDER=workers
AI_MODEL=@cf/meta/llama-3.1-8b-instruct
```

Deploy, and your chat bot is running Llama 3.1 on Cloudflare's edge for free. The free tier covers 10,000 neurons per day — enough for a support widget on most indie projects without ever paying a dollar in inference.

**Where it wins:** latency is low (same network hop as your Worker), data stays inside Cloudflare's infrastructure, and there's nothing to configure. The [5-minute setup guide](/add-live-chat-in-5-minutes) uses Workers AI as the default specifically because you can get a working bot without touching a single API key.

**Where it's limited:** the model catalog is smaller than the big providers. Llama 3.1 8B is good — it handles FAQs, greetings, and triage cleanly — but it will occasionally struggle with nuanced reasoning, very long conversations, or answers that require careful tone. If your chat bot is representing a brand that cares about voice, you'll feel the edges.

## OpenAI: the reliable workhorse

If your team already has an OpenAI key, the switch is two lines:

```env
AI_PROVIDER=openai
AI_MODEL=gpt-4o-mini
OPENAI_API_KEY=sk-...
```

**The cost math:** GPT-4o-mini runs around $0.15 per million input tokens and $0.60 per million output tokens. A typical chat exchange — a greeting, a few turns of Q&A, maybe a handoff message — is roughly 2,000–4,000 tokens total. That's less than $0.001 per conversation. Even at 20,000 conversations a month you're looking at under $20 in inference.

GPT-4o is roughly 30× more expensive per token. For a live chat widget, that's rarely justified — use 4o-mini unless you can point to specific failures that a smarter model would fix.

**Where OpenAI wins:** the widest API ecosystem. Function calling, structured outputs, vision, and the most battle-tested rate limits at scale. If you need the bot to call your own APIs mid-conversation, OpenAI's function-calling interface is the easiest to wire up.

**The honest trade-off:** you're dependent on their pricing and uptime. Krispy's config model means you can swap providers in an afternoon — but that's still a real migration, not zero cost.

## Anthropic: when tone is the constraint

Claude tends to land differently than GPT in one specific way: it follows detailed instructions more precisely and hedges less on things it's uncertain about. For a live chat bot that's representing a brand — one where a hallucinated answer or a weirdly casual response causes actual damage — that reliability is worth paying for.

```env
AI_PROVIDER=anthropic
AI_MODEL=claude-haiku-4-5
ANTHROPIC_API_KEY=sk-ant-...
```

Claude Haiku is the fastest and cheapest Claude model. Pricing is roughly $0.80 per million input tokens and $4 per million output tokens — more expensive than GPT-4o-mini, still cheap in absolute terms for chat volumes. A month of 20,000 conversations probably costs $25–60 depending on your average turn length.

**Where Anthropic wins:** instruction fidelity on detailed prompts. If you've put real effort into your Krispy system prompt — brand voice, escalation rules, what the bot should never say — Claude is more likely to honor all of it across a long conversation, without drifting or padding.

## The practical call

Start with Workers AI. It's already wired into Krispy, it costs nothing, and it's fast. Upgrade when you run into a real problem — not in anticipation of one.

If you upgrade, default to OpenAI GPT-4o-mini. It's cheap, the API is mature, and most chat use cases don't need more. Move to Claude Haiku if you have a complex system prompt, a brand-voice requirement, or users asking questions that require careful, specific answers.

GPT-4o and Claude Sonnet are there for edge cases — not defaults.

## FAQ

**Can I use a different model, like Gemini or Mistral, with Krispy?**

Yes. If your provider exposes an OpenAI-compatible API — Mistral, Groq, Together AI, and local Ollama all do — point `AI_PROVIDER=openai` at a custom base URL. Native adapters for additional providers are on the roadmap; PRs are open.

**Does switching AI models mid-production break anything for active users?**

No. The model config is environment-level — change `AI_MODEL` and redeploy. Krispy's human-in-the-loop design keeps conversation context in your Telegram thread, not inside the model, so there's no session state to migrate. Existing conversations continue normally.

**Is Workers AI reliable enough for a production chat widget?**

For most projects, yes. Llama 3.1 8B handles routine chat cleanly. Where it occasionally falls short is on multi-step reasoning or very specific domain knowledge — which is where a well-written system prompt and the Telegram handoff matter more than the model tier. Let the AI handle volume; let humans handle the hard stuff.

**Does Krispy send my users' messages to Anthropic or OpenAI?**

Only if you configure one of those providers. With Workers AI, messages stay within Cloudflare's network. With OpenAI or Anthropic, messages go to their API under your account's data processing terms — review those agreements if you handle EU user data or have strict privacy requirements.
